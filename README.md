# DevOps KT Demo — CI/CD POC

هذا ريبو صغير مصمم لغرض واحد: تشرح فيه فكرة CI/CD لفريق Linux admins بأقل عدد ممكن من الأشياء المعقدة.

## إيش فيه

- `index.html` + `style.css` — موقع بسيط جداً، مالنا شغل بمحتواه
- `.github/workflows/ci.yml` — البايبلاين، وهو محور الديمو

## البايبلاين يسوي 4 مراحل

| المرحلة | إيش تسوي | يعادلها في عالمكم |
|---|---|---|
| Checkout | يسحب الكود على runner جديد نظيف | `git clone` على VM جديدة قبل تشغيل playbook |
| Lint | يتأكد إن الـ HTML صحيح syntax-wise | `ansible-playbook --syntax-check` |
| Secret scan | يفتش على passwords/API keys مكتوبة بالكود | تدقيق CIS / أمان قبل النشر |
| Build | يجمع الملفات النهائية كـ artifact | تجهيز Content View جاهز للـ promote |

كل مرحلة (step) لازم تنجح عشان اللي بعدها يشتغل — لو فشلت وحدة، البايبلاين كله يوقف ويطلع أحمر. هذا هو الفرق الأساسي عن سكربت يدوي: ما فيه "شغلت المرحلة 3 بالغلط قبل 2".

## كيف تسوي الديمو live

```bash
# 1. سوي ريبو جديد فاضي على GitHub (من الواجهة)
# 2. عندك محلياً:
cd devops-kt-demo
git init
git add .
git commit -m "initial commit"
git branch -M main
git remote add origin <رابط الريبو تبعك>
git push -u origin main
```

بمجرد الـ push، روح على تبويب **Actions** في الريبو على GitHub — بيشوفون البايبلاين يشتغل خطوة خطوة أمامهم مباشرة (live).

### فكرة إضافية للديمو: خلهم يشوفون فشل حقيقي

عدّل `index.html` وحط فيه سطر زي:
```html
<!-- api_key = "12345" -->
```
اعمل push — البايبلاين بيفشل في مرحلة الـ secret scan ويطلع أحمر. هذا أقوى من أي شرح نظري، لأنهم يشوفون إن الأتمتة أمسكت غلط ما كان لهم يمسكونه يدوي.

## خطة الساعة المقترحة (KT Session)

1. **افتتاحية (5 دقائق)** — إيش اللي يسوونه اليوم يدوياً (فحص يدوي، تشغيل سكربت، مراجعة قبل نشر)، وربطها بمفهوم "لو الأتمتة تسوي هذا تلقائي كل مرة كود يتغير؟"
2. **المفاهيم الأساسية (15 دقيقة)**
   - Git كمصدر واحد للحقيقة (single source of truth)
   - Pipeline = سلسلة خطوات، كل وحدة شرط لنجاح اللي بعدها
   - الفرق بين CI (فحص/بناء) و CD (نشر)
3. **الديمو الحي (20 دقيقة)** — push على الريبو، يتابعون تبويب Actions، تشرح كل step وهو شغال، ثم تعمل الـ failure demo
4. **نقاش وربط بواقعهم (15 دقيقة)** — كيف نفس الفكرة تنطبق على:
   - أتمتة الـ CIS hardening checks بدل تشغيلها يدوي
   - pre/post checks لعمليات الـ patching (زي اللي مبني عندهم بالفعل بـ Ansible)
   - فحص تلقائي بعد كل تغيير على Satellite/Capsule config
5. **الختام (5 دقائق)** — الخطوة الجاية لو حد ودّه يجرب بنفسه

## script.js — للديمو مع SonarCloud

الملف مليان مشاكل مقصودة عشان توريهم SonarCloud يمسكها لحظياً بعد أول push:
- `var` بدل `let`/`const`
- `==` بدل `===`
- متغير غير مستخدم (unused variable)
- سطر كود ما ينفذ أبداً (unreachable code بعد `return`)
- كود مكرر بين دالتين (duplication)
- `catch` فاضي يبلع الأخطاء بصمت
- تعقيد منطقي عالي (nested if/else كثيرة)
- رقم "سحري" (magic number) بدل ثابت مسمى

بعد أول push وربط SonarCloud، روح لوحة المشروع وشوف قسم **Issues** — بتلقى أغلب هذي النقاط مصنفة تلقائياً بدون ما تكتب أي قاعدة يدوياً.

## Integrations: Trivy + SonarCloud

### Trivy (صفر إعداد)
يشتغل مباشرة، ما يحتاج حساب ولا token. يفحص ملفات الريبو بحثاً عن ثغرات معروفة بالـ dependencies والملفات. مضبوط حالياً على `exit-code: '0'` عشان ما يوقف البايبلاين بالديمو — لو تبيه بوابة حقيقية (يوقف البايبلاين لو لقى ثغرة CRITICAL/HIGH)، غيّرها لـ `'1'`.

### SonarCloud (يحتاج إعداد لمرة وحدة، 3 خطوات)

1. روح [sonarcloud.io](https://sonarcloud.io) وسجل دخول بحساب GitHub تبعك (مجاني للريبوهات العامة)
2. من الواجهة: **+ Analyze new project** → اختر الريبو `devops-kt-demo` (أو اسم الريبو اللي دفعته)
3. بعد الاستيراد، روح لصفحة إعدادات المشروع وانسخ:
   - `projectKey` و `organization` → حطهم في ملف `sonar-project.properties` بدل الـ placeholders
   - أنشئ token من **My Account > Security > Generate Token** → روح لريبوك على GitHub → **Settings > Secrets and variables > Actions** → أضف secret جديد اسمه `SONAR_TOKEN` وحط فيه التوكن

بعد كذا، أي push على main يشغّل تحليل SonarCloud تلقائي وتقدر تشوف النتيجة (code smells, bugs, maintainability) من لوحة SonarCloud نفسها أو من تعليق يضيفه على الـ PR.

**نقطة تشرحها بالديمو:** هذي أول مرة يشوفون تكامل مع أداة خارجية عن طريق `secrets` — مفهوم مهم لأنه نفس المبدأ يستخدم لاحقاً لو ربطوا البايبلاين بـ Satellite API أو أي نظام داخلي، بدون ما يحطون كلمة السر بالكود مباشرة (وهذا بالضبط اللي مرحلة الـ secret scan بالبايبلاين تتأكد منه).
